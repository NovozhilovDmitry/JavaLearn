## 🔄 **Дженерики (Generics)**

### Что это?
**Дженерики** - это способ создания классов, интерфейсов и методов, которые работают с разными типами данных, сохраняя безопасность типов.

### Простая аналогия:
Представьте коробку. Без дженериков - это просто "коробка", и вы не знаете, что внутри. С дженериками - это "коробка для книг", "коробка для игрушек" - вы точно знаете содержимое.

### Примеры из фреймворка:

```java
// Без дженериков - опасно, нужно приводить типы
List list = new ArrayList();
list.add("текст");
String text = (String) list.get(0); // нужно явное приведение

// С дженериками - безопасно
List<String> list = new ArrayList<>();
list.add("текст");
String text = list.get(0); // автоматически знаем тип
```

**В вашем фреймворке:**
```java
// FormUnit использует дженерик T
public interface FormUnit<T extends FormWin> {
    T itSelf(); // возвращает объект типа T
}

// Использование
public class МояФорма implements FormUnit<МояФорма> {
    public МояФорма itSelf() {
        return this; // возвращает этот же объект с правильным типом
    }
}
```

### Зачем нужны?
- **Безопасность типов** - компилятор ловит ошибки
- **Удобство** - не нужно постоянно приводить типы
- **Переиспользование кода** - один класс работает с разными типами

## 🏷️ **Аннотации (Annotations)**

### Что это?
**Аннотации** - это метки/метаданные, которые добавляют информацию к коду. Они не меняют логику программы, но используются другими инструментами.

### Простая аналогия:
Как стикеры на товарах в магазине: "хрупкое", "скоропортящееся". Сами стикеры ничего не делают, но они сообщают важную информацию.

### Примеры из фреймворка:

```java
@Test // говорит JUnit: это тестовый метод
public void testPayment() {
    // код теста
}

@Deprecated // говорит: этот метод устарел, не используйте
public void старыйМетод() { }

@Step("Создание платежа") // говорит Allure: это шаг в отчете
public void создатьПлатеж() { }
```

### Зачем нужны?
- **Маркировка кода** для фреймворков (JUnit, Allure)
- **Генерация документации**
- **Обработка во время компиляции или выполнения**

## 📝 **Интерфейсы vs Абстрактные классы**

### Интерфейсы

**Что это?** Контракт - "что должно быть сделано", но не "как это сделать".

```java
// Интерфейс - определяет ЧТО нужно сделать
public interface FileChecker {
    File файл();    // любой класс должен реализовать этот метод
    byte[] bytes(); // и этот метод
}

// Класс реализует интерфейс
public class TxtFileChecker implements FileChecker {
    public File файл() {
        // конкретная реализация для TXT файлов
        return new File("test.txt");
    }
    
    public byte[] bytes() {
        // конкретная реализация
        return Files.readAllBytes(файл().toPath());
    }
}
```

### Абстрактные классы

**Что это?** Частично реализованный класс, который можно дополнять.

```java
// Абстрактный класс - уже содержит часть реализации
public abstract class Scenario implements AbstractTest {
    protected abstract void beforeScenario(); // абстрактный метод - нужно реализовать
    
    protected void считывание_эталонных_файлов() {
        // уже реализованный метод - можно использовать как есть
        // или переопределить при необходимости
    }
}

// Конкретный класс наследует абстрактный
public class PaymentScenario extends Scenario {
    protected void beforeScenario() {
        // ОБЯЗАНЫ реализовать абстрактный метод
        // подготовка данных для теста платежей
    }
    
    // метод считывание_эталонных_файлов() уже доступен
}
```

### 📊 **Разница и когда использовать:**

| Критерий | Интерфейс | Абстрактный класс |
|----------|-----------|-------------------|
| **Реализация** | Только сигнатуры методов | Может содержать реализацию |
| **Наследование** | Можно реализовать много интерфейсов | Можно наследовать только один класс |
| **Состояние** | Не может содержать поля (кроме констант) | Может содержать поля |
| **Когда использовать** | Определить контракт для разных классов | Создать базовую реализацию для родственных классов |

**Правило выбора:**
- Используйте **интерфейсы**, когда разные классы должны иметь общее поведение
- Используйте **абстрактные классы**, когда у классов есть общая логика

**В вашем фреймворке:**
```java
// FileChecker - интерфейс, т.к. разные типы файлов должны иметь общий контракт
// Scenario - абстрактный класс, т.к. предоставляет общую логику для всех сценариев
```

## 👀 **Test Watchers**

### Что это?
**Test Watchers** - это "наблюдатели" за тестами в JUnit 5. Они позволяют выполнять код в ключевые моменты жизни теста.

### Простая аналогия:
Как видеокамера наблюдения, которая включается:
- Когда тест начинается
- Когда тест проходит успешно  
- Когда тест падает
- Когда тест пропускается

### Пример из фреймворка:

```java
public class Watcher implements TestWatcher {
    
    // Вызывается перед выполнением теста
    @Override
    public void beforeTestExecution(ExtensionContext context) {
        System.out.println("Тест " + context.getDisplayName() + " начинается");
        // В фреймворке: создание точки восстановления БД
    }
    
    // Вызывается когда тест УСПЕШЕН
    @Override
    public void testSuccessful(ExtensionContext context) {
        System.out.println("Тест " + context.getDisplayName() + " УСПЕШЕН!");
        // В фреймворке: логирование успеха
    }
    
    // Вызывается когда тест ПАДАЕТ
    @Override
    public void testFailed(ExtensionContext context, Throwable cause) {
        System.out.println("Тест " + context.getDisplayName() + " УПАЛ!");
        System.out.println("Причина: " + cause.getMessage());
        // В фреймворке: создание скриншота, закрытие приложений
    }
    
    // Вызывается когда тест ПРОПУЩЕН
    @Override
    public void testDisabled(ExtensionContext context, Optional<String> reason) {
        System.out.println("Тест " + context.getDisplayName() + " ПРОПУЩЕН");
        reason.ifPresent(r -> System.out.println("Причина: " + r));
    }
}
```

### Как использовать в тестах:
```java
@ExtendWith(Watcher.class) // подключаем нашего наблюдателя
public class MyTest {
    
    @Test
    void successfulTest() {
        // этот тест вызовет testSuccessful()
    }
    
    @Test  
    void failingTest() {
        fail("Тест упал!"); // этот тест вызовет testFailed()
    }
    
    @Disabled("Еще не готов")
    @Test
    void disabledTest() {
        // этот тест вызовет testDisabled()
    }
}
```

### Зачем нужны Test Watchers?
- **Логирование** - запись информации о тестах
- **Сбор артефактов** - скриншоты, логи при падении
- **Очистка ресурсов** - закрытие соединений, приложений
- **Восстановление состояния** - откат БД, очистка файлов

**В вашем фреймворке** Watcher используется для:
- Создания точек восстановления БД перед тестом
- Создания скриншотов при падении теста  
- Закрытия клиентских приложений после тестов
- Логирования результатов тестирования
