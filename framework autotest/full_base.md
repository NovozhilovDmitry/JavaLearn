## AbstractTest

### Summary
Интерфейс `AbstractTest` предоставляет базовую функциональность для работы с тестовыми данными в фреймворке автотестов. Он позволяет настраивать пути к каталогам с эталонными файлами на основе имени тестового класса и текущего проекта. Использует статическое хранилище путей для обеспечения доступа к тестовым данным из любого места теста.

### Public методы

#### Методы экземпляра
- **`задать_каталог_тестовых_данных()`** - определяет путь к тестовым данным на основе имени класса и проекта. Должен вызываться в конструкторе тестового класса.

#### Статические методы
- **`testDataDirectory()`** - возвращает путь к каталогу тестовых данных текущего теста (первый элемент из стека)

### Dependencies
- **`base.MainClass.currentProject`** - статическая переменная с именем текущего проекта
- Стандартные Java классы: `Path`, `Paths`, `LinkedList`

### Usage Example
```java
public class PaymentTest implements AbstractTest {
    public PaymentTest() {
        задать_каталог_тестовых_данных();
    }
    
    @Test
    public void testPaymentProcessing() {
        Path testData = AbstractTest.testDataDirectory();
        // Использование тестовых данных из каталога
    }
}
```

### Uncertainties
1. **Формат имен классов** - ожидается, что имя тестового класса содержит номер ТК в формате с буквой "Т" (например, "Т123")
2. **Многопоточность** - статический `LinkedList` может вызвать проблемы при параллельном выполнении тестов
3. **Иерархия тестов** - неясно, как работает наследование и переопределение каталогов тестовых данных
4. **Очистка стека** - отсутствует механизм удаления путей после завершения теста
5. **Кодировка символов** - возможны проблемы с обработкой кириллических символов в именах классов
6. **Проектная структура** - жесткая привязка к пути `build/resources/test/TestData` может быть негибкой

## App

### Summary
Класс `App` предоставляет утилиты для управления процессами автотестового окружения. Он содержит методы для завершения серверных и клиентских процессов, работы с Excel и корректного закрытия АРМ. Класс также включает функционал для предотвращения блокировки системы и управления версиями подсистем.

### Public методы
- **`terminate_servers()`** - завершает процессы серверной части (Java-процессы ASDKO)
- **`terminate_clients()`** - завершает клиентские процессы АРМ
- **`terminate_all()`** - завершает все процессы (клиенты и серверы)
- **`terminate_excel(boolean waitFor)`** - закрывает процессы Excel с опциональным ожиданием
- **`закрыть_АРМ_через_выход()`** - имитирует ручной выход из АРМ через меню
- **`terminate_ARM(Projects окно_арма)`** - завершает конкретный АРМ по проекту
- **`currentSubsystemVersion()`** - возвращает текущую версию подсистемы

### Dependencies
- **`enums.Projects`** - перечисление проектов
- **`forms.general.*`** - компоненты UI для работы с диалогами и таблицами
- **`base.MainClass`** - основной класс с общими методами
- **`base.Base`** - базовый тестовый класс
- **Внешние**: `java.awt.*`, `org.slf4j.Logger`, `java.util.concurrent.ExecutorService`

### Usage Example
```java
// Завершение всех процессов перед тестом
App.terminate_all();

// Закрытие АРМ после теста
App.закрыть_АРМ_через_выход();

// Получение версии подсистемы
String version = App.currentSubsystemVersion();
```

### Uncertainties
1. **Процессы Excel** - наличие двух процессов (excel.exe и wfica32.exe) может быть платформозависимым
2. **Логика определения АРМ** - жесткая привязка к заголовкам окон из конфигурации
3. **Многопоточность** - использование `ExecutorService` в `main()` без явного использования
4. **Перемещение курсора** - назначение периодического перемещения мыши не до конца ясно
5. **Неиспользуемые константы** - `aRM_CreditEdo` объявлена, но не используется в методах

## Base

### Summary
Базовый класс `Base` является основой тестового фреймворка, предоставляя общую инфраструктуру для всех тестов. Он управляет инициализацией тестового окружения, подключением к БД, логированием и видеозаписью тестов, а также обеспечивает корректное завершение работы драйверов.

### Public методы
- **`getInstance()`** - возвращает экземпляр синглтона Base
- **`beforeMainSuite()`** - инициализирует тестовое окружение перед запуском тестов
- **`afterClass11()`** - очистка после каждого теста (закрытие Excel, очистка форм)
- **`closeDrivers()`** - завершает работу драйверов после всех тестов

### Dependencies
- **`bugbusters.modules.*`** - модули фреймворка для Allure, видео, драйверов
- **`com.ibm.rational.test.ft.*`** - IBM Rational Functional Tester
- **`db.JDBCConnect`, `db.sql.MainSql`** - работа с базой данных
- **`enums.Projects`, `enums.TableColumn`** - перечисления
- **`logger.Watcher`, `logger.TGExt`** - системы логирования
- **`utils.DateGeneratorLocal`** - утилиты для работы с датами
- **Внешние**: `com.codeborne.selenide`, `org.junit.jupiter.api`, `org.slf4j.Logger`

### Usage Example
```java
// В базовом классе теста
public class MyTest extends Base {
    @Test 
    public void testExample() {
        // Тест использует общую инфраструктуру Base
    }
}

// Использование в тесте
Base.getInstance().afterClass11();
```

### Uncertainties
1. **Архитектура сервисов** - сложная система регистрации сервисов IBM RFT может быть избыточной
2. **Статическая инициализация** - вызов `beforeMainSuite()` в статическом блоке может быть неочевидным
3. **Множество расширений** - взаимодействие между разными расширениями (Allure, Suite, Video) требует координации
4. **Управление памятью** - статические коллекции (`ARMS`, `map`) могут приводить к утечкам памяти
5. **Именование методов** - `afterClass11()` имеет нестандартное имя без явного назначения
6. **Зависимость от MainClass** - прямое использование `Base.main` без четкой инициализации

## CurrentProjectSearcher

### Summary
Класс `CurrentProjectSearcher` предоставляет утилиты для поиска и классификации тестовых классов по проектам. Он автоматически сканирует файловую систему для построения карты тестовых классов, сгруппированных по проектам, и предоставляет методы для определения принадлежности класса к проекту.

### Public методы
- **`projectOfClass(String classNamePart)`** - возвращает проект по части имени класса (бросает `ProjectNotFoundError` если не найден)
- **`testClassesByProjects`** - статическое поле, содержащее сгруппированные тестовые классы

### Dependencies
- **`bugbusters.modules.helper.utils.FileSearcher`** - поиск файлов в файловой системе
- **`bugbusters.modules.helper.utils.StringUtils`** - утилиты для работы со строками
- **`enums.Projects`** - перечисление проектов
- **Внешние**: `java.io.File`, `java.util.*`, `java.util.function.Predicate`

### Usage Example
```java
// Определение проекта по имени класса
Projects project = CurrentProjectSearcher.projectOfClass("PaymentTest");

// Получение списка классов для проекта
List<String> testClasses = CurrentProjectSearcher.testClassesByProjects.get(project);
```

### Uncertainties
1. **Критерии поиска** - жесткая привязка к путям с содержанием "test" и имени проекта может быть ненадежной
2. **Инициализация** - статический блок может вызвать проблемы производительности при старте
3. **Обновление карты** - отсутствует механизм обновления карты классов при добавлении новых тестов
4. **Кэширование** - результаты поиска кэшируются статически, что может быть проблемой в динамической среде

## FileChecker

### Summary
Интерфейс `FileChecker` предоставляет базовую функциональность для работы с файлами в тестовом фреймворке. Он включает методы для обработки ошибок, выгрузки файлов в шлюзы и работы с содержимым файлов. Интерфейс интегрирован с системой отчетов Allure для прикрепления файлов и логирования шагов тестирования.

### Public методы
- **`writeException(Throwable e)`** - сохраняет исключение в файл и прикрепляет к отчету Allure (активируется через системное свойство `FilesChecker.safetyCheck`)
- **`выгрузить_в_Шлюз(Шлюзы шлюз)`** - выгружает файл в указанный шлюз с копированием через временную директорию
- **`файл()`** - абстрактный метод, возвращающий объект File для проверки
- **`bytes()`** - возвращает содержимое файла в виде byte[] (бросает assertion error при ошибке чтения)

### Dependencies
- **`bugbusters.modules.extensions.allure.*`** - интеграция с Allure для шагов и вложений
- **`enums.Шлюзы`** - перечисление шлюзов для выгрузки файлов
- **`bugbusters.modules.helper.utils.DateGenerator`** - утилиты для работы с датами
- **Внешние**: `org.slf4j.Logger`, `java.nio.file.*`, `org.junit.jupiter.api.Assertions`

### Usage Example
```java
public class MyFileChecker implements FileChecker {
    @Override
    public File файл() {
        return new File("test.xml");
    }
    
    public void processFile() {
        выгрузить_в_Шлюз(Шлюзы.ШЛЮЗ_ПИС);
        byte[] content = bytes();
    }
}
```

### Uncertainties
1. **Условие безопасности** - логика активации через `FilesChecker.safetyCheck` может быть неочевидной
2. **Обработка ошибок** - метод `bytes()` завершает тест при ошибке, что может быть избыточно
3. **Временные файлы** - стратегия очистки временных файлов после выгрузки не определена
4. **Кодировка файлов** - не указана кодировка при записи исключений

## FileDKOeditor

### Summary
Интерфейс `FileDKOeditor` определяет контракт для редакторов файлов в системе ДКО. Он предоставляет методы для сохранения изменений в файлы как в исходный файл, так и в новый файл с указанным именем, а также для завершения редактирования без сохранения.

### Public методы
- **`save()`** - сохраняет изменения в исходный файл
- **`saveAs(String file_name)`** - сохраняет изменения в новый файл с указанным именем
- **`complete()`** - завершает редактирование без сохранения изменений

### Dependencies
- **`base.FileChecker`** - базовый интерфейс для работы с файлами
- **Не имеет внешних зависимостей**

### Usage Example
```java
FileDKOeditor<MyFileChecker> editor = getFileEditor();
editor.saveAs("modified_file.xml");
editor.complete();
```

### Uncertainties
1. **Различие между save и complete** - неясно, какие именно действия выполняются при complete()
2. **Обработка ошибок** - не определено поведение при ошибках сохранения
3. **Состояние редактора** - не указано, можно ли использовать редактор после complete()

## MainClass

### Summary
Основной класс `MainClass` является центральным компонентом фреймворка, предоставляющим полный набор методов для управления тестовым окружением АРМ. Он включает функциональность для запуска/остановки приложений, работы с окнами и меню, управления проектами и шлюзами, проверок операционных дней и пользователей. Класс интегрирован с системами логирования, отчетности Allure и базой данных.

### Public методы (ключевые)

#### Управление приложением
- **`beforeMainClass()`** - запускает АРМ, выполняет вход и инициализирует окружение
- **`afterMainClass()`** - завершает работу с АРМ
- **`сменитьПриложение(Projects project)`** - переключает между проектами АРМ
- **`getTopWin(String windowName)`** - проверяет активность окна

#### Работа с меню и окнами
- **`mainMenu(String Путь)`** - открывает пункт меню по пути
- **`searchWindow(...)`** - ищет окно по различным критериям
- **`waitForAnyWindow(int timeout, String... titles)`** - ожидает любое окно из списка

#### Проверки и утилиты
- **`проверка_оперДня(String дата)`** - проверяет операционный день в АРМ
- **`проверка_задан_пользователь(Пользователи пользователь)`** - проверяет вошедшего пользователя
- **`executeCmd(String[] cmds)`** - выполняет команды в командной строке
- **`isCompleted()`** - ожидает завершения операций

#### Устаревшие методы
Множество методов помечены как `@Deprecated` в пользу `TestObjectFinder` и `TestObjectWaiter`

### Dependencies
- **`bugbusters.modules.*`** - модули фреймворка (Allure, хелперы)
- **`com.rational.test.ft.*`** - IBM Rational Functional Tester
- **`db.sql.MainSql`** - работа с базой данных
- **`enums.*`** - перечисления проектов, шлюзов, пользователей
- **`forms.general.*`** - компоненты UI (диалоги, протоколы)
- **`userInterface.*`** - классы пользовательского интерфейса
- **`utils.gates.GatesCleaner`** - очистка шлюзов

### Usage Example
```java
public class PaymentTest extends Base {
    @Test
    public void testPayment() {
        MainClass main = new MainClass(Projects.CREDIT);
        main.beforeMainClass();
        main.mainMenu("Операции->Платежи");
        main.проверка_оперДня("2024-12-25");
        main.afterMainClass();
    }
}
```

### Uncertainties
1. **Сложность и монолитность** - класс содержит слишком много разнородной функциональности
2. **Устаревшие методы** - множество deprecated методов создают путаницу
3. **Статические состояния** - использование статических полей может вызвать проблемы в параллельных тестах
4. **Обработка ошибок** - смешение различных стратегий обработки исключений
5. **Зависимость от RFT** - тесная связь с IBM Rational Functional Tester ограничивает переносимость
6. **Магические числа** - множество жестко закодированных значений (таймауты, количества попыток)
7. **Логика перезапуска АРМ** - сложная логика перезапуска при ошибках может маскировать реальные проблемы

## MainMenuWin

### Summary
Абстрактный класс `MainMenuWin` предоставляет базовую функциональность для работы с окнами, открываемыми через главное меню АРМ. Он стандартизирует процесс открытия окон через меню с настройкой таймаутов и обработкой исключений. Класс автоматически управляет таймаутами ожидания окон и предоставляет механизм повторных попыток при возникновении ошибок активации.

### Public методы
- **`openWindow(boolean fromMainMenu)`** - открывает окно через главное меню или напрямую (основной метод для использования в конструкторах)
- **`win()`** - возвращает объект Window (реализация интерфейса FormWin)
- **`beforeSearchingForWindow()`** - защищенный метод для переопределения логики между проходом по меню и поиском окна
- **`setTimeoutViaMainMenu()`** - устанавливает сокращенный таймаут для окон, открываемых через меню
- **`setDefaultTimeout()`** - восстанавливает стандартный таймаут окон

### Dependencies
- **`base.MainClass`** - основной класс фреймворка
- **`bugbusters.modules.helper.Helper`** - утилиты для повторных попыток
- **`com.rational.test.ft.WindowActivateFailedException`** - исключения RFT
- **`enums.FindBy`** - перечисление стратегий поиска
- **`env.Environment`** - доступ к настройкам окружения
- **`forms.general.abstractForms.FormWin`** - интерфейс окон форм
- **`userInterface.Window`** - класс окон UI

### Usage Example
```java
public class Шлюзы_Objects extends MainMenuWin {
    public Шлюзы_Objects(boolean fromMainMenu) {
        super("Справочники->ПВВС->Шлюзы", 
              FindBy.Class, "asdco.ARM_KO.FrmGate", 
              FindBy.CaptionText, "Шлюзы");
        openWindow(fromMainMenu);
        // инициализация элементов окна
    }
}
```

### Uncertainties
1. **Логика таймаутов** - жесткие значения 5 попыток и зависимость от конфигурационных свойств
2. **Обработка исключений** - автоматическое завершение АРМ при `WindowActivateFailedException` может быть избыточным
3. **Наследование от MainClass** - потенциальное наследование ненужной функциональности
4. **Инициализация полей** - конструктор требует передачи всех параметров поиска, что может быть негибким

## NewTXT

### Summary
Класс `NewTXT` предоставляет функциональность для работы с текстовыми файлами в тестовом фреймворке. Он поддерживает создание, поиск, сравнение и выгрузку текстовых файлов с различными кодировками и разделителями. Класс интегрирован с системой шлюзов и включает расширенные возможности для поиска и валидации содержимого файлов.

### Public методы
- **`NewTXT(Шлюзы FilePath, String FileName, Расширения_файлов расширение, Разделитель разделитель)`** - конструктор для создания файла
- **`NewTXT(String fileNamePart, Расширения_файлов расширение, Разделитель разделитель, Шлюзы... шлюзы)`** - конструктор для поиска существующего файла
- **`задатьКодировку(Кодировки кодировка)`** - устанавливает кодировку файла
- **`сравнить_содержимое_файлов(Object[][] obj)`** - сравнивает содержимое файла с ожидаемыми данными
- **`найтиПодстрокуВФайле(String искомая_подстрока)`** - ищет подстроку в файле
- **`найтиВсеЗаданныеПодстрокиВФайле(String[] искомые_подстроки)`** - ищет несколько подстрок в файле
- **`выгрузить_файл(Object[][] obj)`** - создает и выгружает файл с данными

### Dependencies
- **`enums.Кодировки`**, **`enums.Расширения_файлов`**, **`enums.Шлюзы`** - перечисления для настройки файлов
- **`utils.FileSearcher`** - утилита для поиска файлов
- **`base.MainClass`** - основной класс фреймворка
- **`io.qameta.allure.Step`** - интеграция с Allure
- **Внешние**: `java.io.*`, `java.nio.file.*`, `java.nio.charset.*`, `org.slf4j.Logger`

### Usage Example
```java
NewTXT txt = new NewTXT(Шлюзы.ШЛЮЗ_ПИС, "data", Расширения_файлов.TXT, NewTXT.Разделитель.Табуляция);
Object[][] data = {
    {"Поле1", "Поле2"},
    {"Значение1", "Значение2"}
};
txt.выгрузить_файл(data);
txt.найтиПодстрокуВФайле("Значение1");
```

### Uncertainties
1. **Разделители** - поддержка только табуляции, отсутствие других распространенных разделителей
2. **Производительность** - загрузка всего файла в память может быть проблемой для больших файлов
3. **Кодировки по умолчанию** - Windows-1251 жестко закодирована в конструкторе поиска

## XlsChecker

### Summary
Устаревший класс для проверки данных в XLS-файлах. Осуществляет поиск файлов по шаблону имени и шлюзам, проверяет соответствие данных ожидаемым значениям в виде массивов или блоков. Поддерживает гибкую настройку проверяемых строк и колонок, включая работу с диапазонами и нестрогое сравнение.

### Public методы
1. **XlsChecker(String FileNamePart, String SheetName, Шлюзы... шлюзы)**  
   Конструктор - инициализирует проверщик с поиском файла по шаблону в указанных шлюзах.

2. **getFileName()**  
   Возвращает полный путь к найденному файлу.

3. **checkXslData(String reference, String expectedValue)**  
   Проверяет значение конкретной ячейки. Параметры: reference - адрес ячейки, expectedValue - ожидаемое значение.

4. **getRowCount()**  
   Возвращает количество строк в листе.

5. **setColumns(String letters)**  
   Задает колонки для проверки в формате "A,C-F". Поддерживает диапазоны.

6. **setIndexes(String indexes)**  
   Задает строки для проверки в формате "1,3-6". Поддерживает диапазоны.

7. **checkByArray(Object[][] array, String[][]... blocks)**  
   Основной метод проверки данных по массиву и блокам. Может вызывать Assertion.fail() при несоответствиях.

8. **получитьИндексаНайденныхСтрок(String[][] ПервыеИзСдвоенныхСтрок)**  
   Находит индексы строк, содержащих заданные данные. Возвращает ArrayList<ArrayList<Integer>>.

9. **проверитьСвязанныеСтрокиЭксель(String[][] ПервыеИзСдвоенныхСтрок, String[][] ДанныеСвязанныхСтрок)**  
   Проверяет связность строк (предполагается, что вторые строки следуют за первыми).

10. **setStrict(boolean Strict)**  
    Переключает режим строгого сравнения (полное соответствие) / нестрогого (содержание).

11. **deleteFile()**  
    Вызывает очистку скачанных форм (autoClearDownloadedForms).

12. **найтиЗаданныеПодстроки(String[] stringsForFind, int[] rows)**  
    Ищет подстроки в указанных строках, требует единичного вхождения каждой.

13. **найтиЗаданнуюСтрокуИзНесколькихЯчеек(String[][] stringsForFind, int[] rows)**  
    Аналогично предыдущему, но для строк из нескольких ячеек.

14. **addBlock(String[][] строки)**  
    Добавляет блок данных для проверки.

15. **addBlocks(String[][][] блоки)**  
    Добавляет несколько блоков данных.

### Dependencies
- Apache POI (HSSFWorkbook, HSSFSheet) - работа с Excel
- SLF4J - логирование
- Allure Framework (@Step)
- Внутренние утилиты: FileSearcher, DateGenerator
- JUnit 5 (Assertions.fail)
- Кастомные enum: Шлюзы, Расширения_файлов

### Usage Example
```java

    .setIndexes("1,3")
    .setColumns("A,C")
    .addBlock(new String[][]{{"t1", "t2"}, {"", "t4"}})

```

### Uncertainties
1. Назначение MainClass и метод autoClearDownloadedForms() не определены в коде
2. Логика преобразования чисел и дат в преобразованиеСтроки() может быть хрупкой
3. Неясна точная логика сравнения блоков в checkByArray()
4. Использование кириллических идентификаторов усложняет чтение кода
5. Назначение метода closeExcel() не реализовано (закомментировано)
6. Потенциальные проблемы с кодировками в файлах

# YMetaJson - Документация

## Summary
Класс `YMetaJson` представляет DTO для файла метаинформации `y.meta.json` в составе интеграционного конверта ПИС 1Х. Он содержит структурированные данные о маршрутизации, приоритете, вложенных файлах и дополнительных атрибутах сообщения. Класс предоставляет методы для удобного поиска и доступа к компонентам конверта, включая определение направления сообщения (входящее/исходящее).

## Public методы

### Основные методы
- **`isIncoming()`** - проверяет, является ли конверт входящим (по наличию "-ASDKO" в routeInstanceId)
- **`findContent(Predicate<PisContent> condition)`** - ищет контент по пользовательскому условию
- **`findContent(Type type)`** - ищет контент по типу (DOCUMENT, ESODRECEIPT, SIGN, STATUSINFO)

### Геттеры/сеттеры
Для всех основных полей: `baseSystemId`, `routeInstanceId`, `srcMsgId`, `tid`, `priority`, `content`, `extraAttributes`

### Методы вложенных классов
**`Protocol.Sender`/`Receiver`:**
- **`optionalFeatureByCode(String code)`** - ищет характеристику по коду (Optional)
- **`featureByCode(String code)`** - ищет характеристику по коду (бросает RuntimeException если не найдено)
- **`get(String code)`** - возвращает значение характеристики по коду

## Dependencies
- **`com.google.gson.annotations.SerializedName`** - аннотации для JSON сериализации
- Стандартные Java утилиты: `List`, `Optional`, `Predicate`, `Objects`

## Usage Example
```java
YMetaJson meta = gson.fromJson(jsonContent, YMetaJson.class);

// Проверка направления
if (meta.isIncoming()) {
    // Обработка входящего сообщения
}

// Поиск документа
Optional<YMetaJson.PisContent> document = meta.findContent(YMetaJson.Type.DOCUMENT);

// Получение ИНН отправителя
String inn = meta.getExtraAttributes().getProtocol().getSender().get("INN");
```

## Uncertainties
1. **Логика определения направления** - проверка на "-ASDKO" может быть ненадежной при изменении naming convention
2. **Назначение полей** - точное назначение `baseSystemId`, `routeInstanceId`, `tid` требует дополнительной документации
3. **Статус destinationId** - поле закомментировано, но упоминается в комментариях
4. **Валидные значения** - допустимые значения для `protocolType`, кодов характеристик (INN, OGRN и т.д.)
5. **Обработка получателей** - методы `getReceiver()`/`setReceiver()` работают только с первым получателем из списка
6. **Обязательность полей** - какие поля являются обязательными в реальных сценариях
