# AbstractTest - Документация

## Summary
Интерфейс `AbstractTest` предоставляет базовую функциональность для работы с тестовыми данными в фреймворке автотестов. Он позволяет настраивать пути к каталогам с эталонными файлами на основе имени тестового класса и текущего проекта. Использует статическое хранилище путей для обеспечения доступа к тестовым данным из любого места теста.

## Public методы

### Методы экземпляра
- **`задать_каталог_тестовых_данных()`** - определяет путь к тестовым данным на основе имени класса и проекта. Должен вызываться в конструкторе тестового класса.

### Статические методы
- **`testDataDirectory()`** - возвращает путь к каталогу тестовых данных текущего теста (первый элемент из стека)

## Dependencies
- **`base.MainClass.currentProject`** - статическая переменная с именем текущего проекта
- Стандартные Java классы: `Path`, `Paths`, `LinkedList`

## Usage Example
```java
public class PaymentTest implements AbstractTest {
    public PaymentTest() {
        задать_каталог_тестовых_данных();
    }
    
    @Test
    public void testPaymentProcessing() {
        Path testData = AbstractTest.testDataDirectory();
        // Использование тестовых данных из каталога
    }
}
```

## Uncertainties
1. **Формат имен классов** - ожидается, что имя тестового класса содержит номер ТК в формате с буквой "Т" (например, "Т123")
2. **Многопоточность** - статический `LinkedList` может вызвать проблемы при параллельном выполнении тестов
3. **Иерархия тестов** - неясно, как работает наследование и переопределение каталогов тестовых данных
4. **Очистка стека** - отсутствует механизм удаления путей после завершения теста
5. **Кодировка символов** - возможны проблемы с обработкой кириллических символов в именах классов
6. **Проектная структура** - жесткая привязка к пути `build/resources/test/TestData` может быть негибкой
